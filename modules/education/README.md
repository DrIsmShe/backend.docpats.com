# modules/education — подготовка к экзаменам

Модуль подготовки врачей к экзаменам: сертификация, резидентура, международные
сертификаты, НМО, внутреннее обучение персонала клиники. Смонтирован в
`index.js` как `/api/v1/education`.

## Почему модуль глобальный, а не подмодуль clinic

`modules/clinic/*` — мультитенантные: каждый запрос проходит через
`tenantMiddleware`, а `tenantScoped`-плагин подмешивает `clinicId` в каждый
запрос к базе. Здесь этого **нет намеренно**: аудитория модуля — врачи и
резиденты, которые могут не состоять ни в одной клинике. Если убрать банк
вопросов под тенант, его нельзя будет продавать отдельно от клиники.

Клиника появляется в модуле в двух местах, и оба опциональны:

- `ExamProgram.ownerClinicId` — приватная программа клиники (`null` = публичная);
- `ExamAttempt.clinicId` — попытка в рамках назначенного клиникой курса.

Из этого вырастет будущий `modules/clinic/clinic-training/` — тонкий
tenant-scoped слой (назначения, дедлайны, сертификаты, отчёт «кто прошёл
HIPAA-тренинг»). Ресурс `education` в `common/auth/permissions.js` появится
вместе с ним; пока RBAC делает `middlewares/educationAuth.js` — своя лестница
ролей поверх `session.userId`, потому что `common/auth/can.js` требует
`ClinicMembership`, которого у учащегося может не быть.

Лестница ролей:

| | Кто | Что может |
| --- | --- | --- |
| `requireLearner` | любой авторизованный | проходить тесты |
| `requireAuthor` | `admin` | создавать вопросы, загружать файлы на импорт |
| `requireReviewer` | `admin` | публиковать, принимать и отклонять ревью |

Весь редакторский контур закрыт ролью `admin` — включая врачей. Когда
появятся внешние авторы-врачи, работающие под ревью админа, достаточно
добавить `doctor` в `AUTHOR_ROLES`: разделение author/reviewer сохранено
именно ради этого.

## Подмодули

| Путь | За что отвечает |
| --- | --- |
| `education-catalog/` | `ExamProgram`: страна, регион, тип экзамена, орган, blueprint тем с весами, привязка к рубрике, деление на блоки |
| `education-categories/` | `ExamCategory`: рубрикатор витрины (категория → подкатегория), который создаёт админ |
| `education-items/` | `ExamItem`: банк вопросов, редакторский цикл, версии, item analysis |
| `education-attempts/` | `ExamAttempt`: сборка сессии, приём ответов, скоринг, готовность |
| `education-ingest/` | `ExamImportJob`: импорт вопросов из PDF/изображений (в т.ч. через ИИ) |

Общие справочники — `constants.js`. Enum'ы моделей, zod-валидаторы и фильтры
сервисов ссылаются на него; рассинхрон здесь ломает сразу все четыре подмодуля.

## Экзамены «по всем странам»

`ExamProgram` описывается четырьмя независимыми осями, поэтому витрину можно
резать как угодно:

- `country` — ISO 3166-1 alpha-2 (`TR`, `SA`, `AZ`, `RU`) либо `INT` для
  наднациональных экзаменов;
- `region` — `cis` / `europe` / `mena` / `asia` / `africa` / `americas` /
  `oceania` / `international`;
- `examType` — `licensing`, `residency_entrance`, `board_certification`,
  `international_certificate`, `cme`, `university`, `internal_training`;
- `authority` — орган, проводящий экзамен (ÖSYM, SCFHS, DHA, ECFMG).

`GET /programs/countries` возвращает агрегат для навигации: сколько программ в
каждой стране и какие типы экзаменов там есть.

## Рубрикатор витрины — категории создаёт админ

Оси `country/region/examType` фиксированы кодом. Поверх них лежит **свободный
рубрикатор**, который наполняет админ, — `ExamCategory` (коллекция
`exam_categories`), прямой аналог категорий статей. Ничего не задано заранее:
пустой каталог — нормальное стартовое состояние.

- дерево **произвольной глубины**: категория → подкатегория → под-подкатегория
  → … (`parentId`, self-ref). Ограничение одно — запрет циклов: категорию
  нельзя вложить в саму себя или в своё же поддерево (`assertParentValid`
  идёт вверх по цепочке предков предполагаемого родителя). Витрина ходит по
  дереву как по папкам, с хлебными крошками;
- `ExamProgram.categoryId` привязывает тест к любому узлу на любом уровне;
  `null` = тест лежит в корне витрины («Все тесты»);
- `GET /categories` отдаёт дерево с числом тестов (`programCount` у узла =
  свои тесты + тесты всего поддерева, считается рекурсивно). `scope=all`
  считает все статусы (админка), по умолчанию — только опубликованные
  публичные (витрина);
- удаление рубрики **не трогает тесты**, только их привязку, поэтому оно
  блокируется, пока к категории привязаны тесты или у неё есть подкатегории —
  сначала перенеси их. Создание/правка — `requireAuthor`, удаление —
  `requireReviewer`.

## Большой экзамен блоками по 50

Экзаменационные сборники доходят до 2000 вопросов; проходить их разом
бессмысленно. `ExamProgram.blockSize` (например, 50) включает деление теста на
**блоки** — по `blockSize` опубликованных вопросов в каждом.

- блок — это **детерминированный срез**, а не выборка по темам: вопросы
  сортируются по `(createdAt, _id)` и режутся по `blockSize`. Одна и та же
  сортировка используется в `getProgramBlocks` (что показывает витрина) и в
  `startAttempt` при `blockIndex` — поэтому «Блок 2» там и там один и тот же
  набор;
- `GET /programs/:id/blocks` возвращает `{ blockSize, totalCount, blocks:
  [{ index, from, to, count }] }`; последний блок короче остальных;
- попытка по блоку запускается с `blockIndex` и режимом `tutor`/`timed`
  (`mock`/`drill` к блоку неприменимы — они собирают состав по blueprint);
- «висящая» попытка блокирует новую **точечно на каждый блок**, а не на весь
  тест: блоки проходятся независимо. `ExamAttempt.blockIndex` (0-based, `null` =
  попытка по всему тесту) хранит, к какому блоку относится сессия.

## Архив против удаления

`DELETE /programs/:id` — **мягкий архив** (`status: archived`), обратимо: тест
исчезает из витрины и списков, но вопросы и попытки целы.

`DELETE /programs/:id/hard` — **удаление насовсем** вместе с банком вопросов.
Оно запрещено, если по тесту уже есть попытки (`ConflictError`,
`details.reason = "has_attempts"`): история сдач ценна для готовности и item
analysis, поэтому «Удалить» никогда не стирает молча чужой прогресс — админу
предлагается архив. Оба закрыты `requireReviewer`.

## Blueprint — не украшение

`ExamProgram.blueprint` — карта тем с процентными весами. На ней держится
две вещи:

1. **Состав пробного экзамена.** `mode: "mock"` набирает вопросы строго по
   весам (метод наибольшего остатка, чтобы округление не теряло вопросы).
2. **Готовность к экзамену.** `GET /programs/:id/readiness` взвешивает
   результаты по темам теми же весами, что и настоящий экзамен. Отличная
   неврология не компенсирует проваленную кардиологию, если кардиология
   весит 80%.

Раздел blueprint нельзя удалить, пока на него ссылаются активные вопросы —
иначе скоринг по темам осиротеет.

## Происхождение контента — юридически значимое поле

Каждый вопрос обязан нести `source.kind`:

| Значение | Смысл |
| --- | --- |
| `original` | Написан авторами проекта |
| `public_government` | Официальный открыто опубликованный материал, разрешённый для подготовки (обязателен `authority`) |
| `licensed` | Получен по лицензии (обязателен `licenseNote`) |
| `ai_generated` | Черновик, сгенерированный моделью |

Копировать вопросы чужих коммерческих банков нельзя. Гейты в
`assertPublishable` не дают опубликовать заимствованный материал без указания
органа и условий использования.

## Единственный путь вопроса к учащемуся

```
draft → in_review → published
         ↓
      rejected → (правка) → in_review
```

- `createItem` **всегда** создаёт `draft`, даже если в теле запроса пришёл
  `status: "published"`; импорт сразу переводит созданный вопрос в
  `in_review` — оставшийся черновиком вопрос не виден ни в очереди, ни
  учащимся, то есть нигде;
- `published` присваивается **только** внутри `reviewItem()`, а этот роут
  закрыт `requireReviewer`;
- правка содержания опубликованного вопроса не мутирует его на месте: создаётся
  новая версия со ссылкой `previousVersionId`, старая архивируется, новая идёт
  на ревью заново. Прошлые попытки остаются воспроизводимыми.

### Что блокирует публикацию, а что нет

`assertPublishable` разделяет две разные вещи.

**Запрещают публикацию** — корректность и права:

- нет правильного ответа;
- у заимствованного материала не указан орган (`public_government`, `licensed`);
- у лицензионного не описаны условия использования;
- машинно-извлечённый вопрос не проверен человеком (ниже).

**Не запрещают, но попадают в замечания** (`collectQualityWarnings`):
отсутствие общего объяснения, объяснений к вариантам, источников.

Объяснения намеренно необязательны. Государственные сборники сплошь идут
без разборов, и жёсткое требование запретило бы импорт как класс. Наличие
разбора — вопрос качества банка, а не его корректности: решает редактор,
а система показывает пробелы и пишет их в лог при публикации, чтобы позже
было видно, какую часть банка нужно дописать.

### Гейт машинно-извлечённого контента

`assertPublishable` требует ревью человеком по **двум независимым** признакам:

- `source.kind === "ai_generated"` — содержание **создано** моделью;
- `importJobId != null` — содержание **извлечено** моделью из файла.

Второй признак существует потому, что официальный государственный тест,
распознанный из PDF, имеет безупречный `source.kind = "public_government"` —
но распознавание могло потерять вариант ответа или перепутать ключ.

## ИИ-импорт из PDF и изображений

Поток:

```
файл (в памяти) → экстрактор → draftItems в ExamImportJob
                                     ↓ правка оператором
                              ExamItem (draft) → ревью → published
```

Черновики лежат **внутри задания импорта**, а не сразу в банке вопросов:
распознавание ошибается, и мусор из колонтитула не должен попадать в
`exam_items` и портить статистику и очередь ревью.

### Форматы файлов

Anthropic API принимает напрямую только PDF, изображения и текст. Всё
остальное приводится к тексту на нашей стороне — `extractors/fileToText.js`.

| Формат | Как обрабатывается |
| --- | --- |
| PDF | `document`-блок, base64 |
| JPG, PNG, WebP, GIF | `image`-блок, base64 |
| DOCX | `mammoth.extractRawText` → текст |
| TXT, MD, CSV, TSV, JSON, XML | декодирование → текст |
| HTML | снятие тегов → текст |
| RTF | снятие управляющих слов → текст (best-effort) |

Осознанно **не** поддерживаются: `.doc`, `.xls/.xlsx`, `.ppt/.pptx`, `.odt`,
архивы, DjVu, EPUB, TIFF, HEIC. На каждый — конкретный совет вместо общего
отказа («сохраните лист как CSV», «экспортируйте презентацию в PDF»):
пересохранить файл почти всегда быстрее, чем искать обходной путь.

Excel читается через CSV намеренно. Библиотека `exceljs` тянет за собой
зависимость с известной уязвимостью, и для HIPAA-платформы это плохой
размен ради формата, который в тестовых сборниках встречается редко.

**Тип файла определяется по MIME И по расширению.** Браузеру верить нельзя:
Windows отдаёт `.csv` как `application/vnd.ms-excel`, `.md` часто приходит
с пустым типом, а файл из архива — как `application/octet-stream`. Поэтому
`fileTypes.js` смотрит на оба признака, а enum MIME-типов в модели убран.

**Кодировка определяется, а не предполагается.** Русские методички
регулярно приходят в windows-1251; молча прочитать их как UTF-8 значит
скормить модели «Ð£ÐšÐÐ–Ð˜Ð¢Ð•» вместо вопросов. `decodeText` проверяет BOM,
затем UTF-8, затем откатывается на cp1251.

Экстракторы выбираются как платёжные провайдеры — через env:

```bash
EDUCATION_EXTRACTOR=manual   # по умолчанию: вопросы приходят уже разобранными
EDUCATION_EXTRACTOR=claude   # ИИ-извлечение, требует ANTHROPIC_API_KEY
EDUCATION_EXTRACTOR_MODEL=claude-opus-4-8   # необязательно
```

`manual` — умышленный дефолт: ИИ-импорт стоит денег и отправляет файл во
внешний сервис, такое поведение должно включаться осознанно.

### Структура теста строится из файла

Чтобы админ мог просто загрузить файл, ничего не размечая заранее, модель
возвращает не только вопросы, но и `suggestedProgram` — предложенное
название теста и разделы с весами, выведенные из содержимого файла.

`runExtraction` применяет это предложение **только если у программы ещё нет
blueprint**. Если темы уже размечены руками, предложение игнорируется
целиком: чужая разметка поверх ручной сломала бы и скоринг, и привязку уже
существующих вопросов. Веса нормализуются к 100 — модель ошибается в
арифметике чаще, чем в содержании, и падать из-за суммы 99.7 было бы глупо.

Техническое название вида «Черновик импорта: …» модель заменяет на
осмысленное; название, введённое человеком, не перетирается.

`claude.extractor.js` отдаёт PDF как `document`-блок, изображение — как
`image`-блок, и получает ответ через structured outputs, поэтому форма JSON
гарантирована схемой, а не разбирается эвристикой. Системная инструкция
запрещает главное: **извлекай, не сочиняй**. Если ключ ответа в файле не
указан, `correctKeys` приходит пустым и оператор проставляет его руками — модель
не угадывает по своим медицинским знаниям.

**Исходный файл на сервере не сохраняется** — он живёт в памяти ровно на время
извлечения. Это и меньше копий лицензионных материалов, и на один рубеж утечки
меньше. Нужен оригинал для сверки — оператор указывает внешний `file.url`.

## Режимы прохождения

| Режим | Таймер | Разбор | Состав |
| --- | --- | --- | --- |
| `tutor` | нет | сразу после каждого ответа | случайный / по темам |
| `timed` | да | в конце | случайный / по темам |
| `mock` | да | в конце | строго по blueprint |
| `drill` | нет | в конце | три худшие темы учащегося |

Правильные ответы **не отдаются клиенту до ответа**: `toLearnerView()` без
`includeAnswer` вырезает `correctKeys`, `explanation` и объяснения по вариантам.
Проверка ответа делается только на сервере.

Неотвеченный вопрос при сдаче считается неверным — счёт идёт по составу
попытки, а не по числу ответов, иначе можно «набрать 100%», ответив на один
вопрос из шестидесяти.

## Item analysis

`GET /programs/:programId/item-analysis` — отчёт редактору по опубликованным
вопросам с достаточной статистикой:

- `pValue < 0.25` — почти никто не отвечает верно: вероятно, вопрос некорректен
  или ключ ответа неверный;
- `pValue > 0.95` — не различает уровни подготовки, балласт;
- «мёртвый дистрактор» — вариант, который не выбрал никто.

Это то, что отличает серьёзный банк вопросов от свалки.

## Экраны на клиенте

Контур учащегося и контур управления разведены по разным зонам приложения.

**Учащийся** — обычные страницы, доступны любому авторизованному:

| Маршрут | Что делает |
| --- | --- |
| `/education` | Витрина: навигация по категориям/подкатегориям, поиск, сетка тестов |
| `/education/programs/:id` | Готовность по темам, блоки, запуск попытки, история |
| `/education/attempts/:id` | Тренажёр и разбор результата |

**Управление** — только админ-панель, роль `admin`:

| Маршрут | Что делает |
| --- | --- |
| `/admin/education-programs` | Список тестов: публикация, переименование, привязка к категории, размер блока, архив, удаление насовсем |
| `/admin/education-categories` | Рубрикатор: создание/правка/удаление категорий и подкатегорий (как категории статей) |
| `/admin/education-import` | Загрузка файла, чистка распознанного, перенос в банк |
| `/admin/education-review` | Ревью: подтвердить ответ, при желании дописать разбор |

**Публикаций две, и это не избыточность.** Опубликованный вопрос попадает
в банк программы; учащиеся увидят тест только когда опубликована и сама
программа (`ExamProgram.status = "published"`). Вопросы можно накапливать
месяцами, а тест открыть один раз, когда он готов — поэтому решения
разделены.

Ссылок из пользовательской зоны в админскую нет намеренно. `AdminLayout`
проверяет права через `GET /admin/admin-panel`, а бэкенд закрывает те же
маршруты `requireAuthor`/`requireReviewer` — обе проверки независимы, и
вторая является настоящей.

Отдельного `<ProtectedRoute>` в проекте нет — страницы учащегося
самозащищаются, как и остальные зоны: 401 от API уводит на `/login`.

Экран ревью специально устроен как «прочитай и дополни»: поля объяснений
редактируются прямо в карточке, а кнопка публикации заблокирована, пока
чего-то не хватает, с перечислением чего именно. Кнопка «Опубликовать»
сначала сохраняет правки и только потом отправляет решение — иначе гейт
отверг бы публикацию из-за объяснений, которые существуют только на экране.

Сид-скрипт демо-программы: `node scripts/seed-education.mjs` (флаги
`--wipe`, `--remove`).

нормализация распознанного и перенос черновиков в банк, а также рубрикатор,
жёсткое удаление и деление на блоки (`categories.test.js`: дерево категорий и
его защиты, guard удаления теста с попытками, детерминированный срез блока,
независимость блоков). ИИ-экстрактор в тестах не вызывается (ходит во внешний
API) — проверяется всё, что от него не зависит.

```bash
npx vitest run __tests__/education
```
